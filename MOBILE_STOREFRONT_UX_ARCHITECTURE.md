## 1. لماذا Company First؟

### كيف يفكر العميل فعلياً؟

```
العميل لا يقول:             "أريد المنتج 1056"
العميل يقول:                "افتح منتجات إيفا"
ثم:                         "أضف 5 زبادي، 3 حليب"
ثم:                         "روح لشركة ستارفيل"
ثم:                         "أضف 10 مكرونة، 4 صلصة"
ثم:                         "تم، أنشئ الطلب"
```

### المشكلة في Product First الحالي

```
الوضع الحالي:               الشركات ← اختيار شركة ← منتجاتها ← إضافة ← رجوع للشركات
                            ← شركة أخرى ← منتجاتها ← إضافة ← رجوع للشركات
                            ← السلة
                            
هذا مسار طويل جداً! المستخدم يضطر للرجوع للخلف بين كل شركة وأخرى.
```

### الحل: Company First

```
الوضع الجديد:
  الرئيسية ← اختيار شركة ← منتجاتها + إضافة ← (يبقى في نفس الشاشة)
           ← اختيار شركة أخرى من القائمة الجانبية ← منتجاتها + إضافة
           ← السلة ← إنشاء طلب

الشركات متاحة دائماً من Bottom Nav (التبويب الثاني)
المنتجات دائماً داخل سياق الشركة
السلة مجمعة حسب الشركة مع مجاميع فرعية
```

### الفرق الجوهري

| الجانب | Product First (الحالي) | Company First (الجديد) |
|--------|----------------------|------------------------|
| **التبويب الرئيسي** | المنتجات | **الشركات** |
| **نقطة الدخول** | ابحث عن منتج | **اختر شركة** ثم تصفح منتجاتها |
| **السلة** | قائمة منتجات | **مجمعة حسب الشركة** بمجاميع فرعية |
| **التنقل بين الشركات** | رجوع للخلف ← شركة أخرى | **قائمة جانبية** أو Bottom Nav مباشر |
| **البحث** | يبحث عن منتجات | **يبحث عن شركات + منتجات** (الشركات أولاً) |
| **الرئيسية** | أحدث المنتجات | **شبكة الشركات** كبوابة رئيسية |

## 2. قراران معماريان

### 2.1 مبدأ المسارين

كما تم اعتماد سابقاً:

```
المسار الأول (فوري):        Storefront + Field — Mobile First
المسار الثاني (مؤجل):       OPS — Desktop First، الجداول باقية
```

### 2.2 محرك البحث الموحد (#search)

كما تم اعتماد سابقاً، **#search** هو المرجع الوحيد. مع Company First، سنضيف الشركات كنوع قابل للبحث في نتائج `#search` (راجع [القسم 6](#6-نظام-البحث-الموحد)).

### 3.2 اختصارات سريعة في الرحلة

| الموقف | الإجراء المباشر |
|--------|----------------|
| **يعرف اسم الشركة** | ابحث في #search ← اختر الشركة ← منتجاتها |
| **يعرف المنتج والشركة** | ابحث في #search ← اختر المنتج ← يظهر مع الشركة |
| **يريد شراء متكرر** | الرئيسية ← آخر طلب ← أضف الكل للسلة |
| **عنده صفقة اليوم** | الرئيسية ← بنر 🔥 ← منتجات الصفقة ← أضف ← ارجع للشركات |
| **مندوب في زيارة** | تفاصيل الزيارة ← إنشاء طلب ← اختيار شركة ← أصناف |

### 5.2 Storefront — شاشات العميل

| # | الشاشة | الملف الحالي | ما الذي يتغير |
|---|--------|-------------|---------------|

## 6. نظام البحث الموحد

### 6.1 التحليل: الوضع الحالي

لدينا حالياً:

| الطريقة | الملف | ماذا يبحث |
|---------|-------|-----------|
| **#search page** | `search.js` + `searchApi.globalSearch()` | منتجات، عملاء، موظفين عبر RPC |
| **Global Search bar** | `globalSearch.js` | عملاء، فواتير، منتجات، مناديب — 4 استعلامات منفصلة |
| **OPS products** | `productsApi.js` | منتجات فقط |
| **OPS employees** | `employeeGovernanceApi.js` | موظفين فقط |

**المشكلة:** لا يوجد بحث عن **الشركات** في أي من هذه الطرق. العميل لا يستطيع البحث عن شركة والحصول على نتائج.

### 6.2 إضافة الشركات إلى البحث الموحد

#### ما يتغير في searchApi.js

```javascript
// إضافة دالة جديدة (لا نغير الموجود)
export async function searchCompanies(query) {
  if (!query || query.trim().length < 2) return [];
  const q = query.trim();
  try {
    const r = await fetch(`${API}/companies?select=id,company_name,company_code,company_logo_url&is_active=eq.true&or=(company_name.ilike.*${encodeURIComponent(q)}*,company_code.ilike.*${encodeURIComponent(q)}*)&limit=5`, { headers: _headers() });
    if (r.ok) return r.json();
  } catch { /* ignore */ }
  return [];
}

// دالة موحدة جديدة
export async function unifiedSearch(query) {
  const [global, companies] = await Promise.all([
    globalSearch(query),    // (موجود — منتجات، عملاء، موظفين)
    searchCompanies(query), // (جديد — شركات)
  ]);
  // دمج النتائج: الشركات أولاً
  return {
    companies,           // شركات ← تظهر أولاً في النتائج
    products: global.filter(r => r.entity_type === 'product'),
    customers: global.filter(r => r.entity_type === 'customer'),
    employees: global.filter(r => r.entity_type === 'employee'),
  };
}
```

#### ما يتغير في مكون البحث

```
old:    globalSearch.js — 4 استعلامات منفصلة

new:    Reusable Search Component
        يستخدم unifiedSearch() داخلياً
        يعرض النتائج مرتبة: شركات ← منتجات ← عملاء ← موظفين
```

### 6.3 خريطة توحيد الشاشات

| الشاشة | الحالي | الجديد — Unified Search |
|--------|--------|--------------------------|
| **الرئيسية** | بحث يحول لـ #search?q= | `<reusable-search>` مع نتائج فورية |
| **الشركات** | لا بحث | `searchCompanies()` — بحث في أسماء الشركات |
| **منتجات الشركة** | لا بحث داخلي | `searchByType('product', q)` + فلترة `company_id` |
| **منتجات (عام)** | `productsApi.js` (ilike) | `searchByType('product', q)` |
| **السلة (إضافة)** | قائمة منتجات | `searchByType('product', q)` + شركة السياق |
| **العملاء** | قائمة مباشرة | `searchByType('customer', q)` |
| **الطلبات** | فلترة | `searchByType('invoice', q)` |
| **Global Search bar** | 4 استعلامات منفصلة | `unifiedSearch()` موحد |
| **OPS المنتجات** | `productsApi.js` | `searchByType('product', q)` |
| **OPS العملاء** | قائمة | `searchByType('customer', q)` |
| **OPS الموظفين** | `employeeGovernanceApi.js` | `searchByType('employee', q)` |

### 6.4 ما لا نفعله

```
❌ لا نغير RPC global_search الموجود
❌ لا نغير search_index_runtime
❌ لا نغير Ranking الحالي
❌ لا نضيف فهارس جديدة
❌ لا نعيد كتابة منطق البحث
❌ لا نضيف مكتبة بحث خارجية
```

نحن فقط:
- نضيف `searchCompanies()` — بحث ilike بسيط على جدول companies (موجود فعلاً)
- ننشئ `unifiedSearch()` — دمج نتائج الـ RPC + نتائج الشركات
- ننشئ `Reusable Search Component` — واجهة موحدة مع ترتيب Company First للنتائج

## 7. الشرائح والعروض مع Company First

### 7.1 مبدأ الظهور

**مع Company First، العرض والشرائح يظهران ضمن سياق الشركة.**

#### قواعد عرض السعر في كل كرت منتج

| الحالة | ما يظهر في الكرت |
|--------|------------------|
| **لا شريحة، لا عرض** | `السعر: 960 ج.م` — سعر واحد فقط |
| **شريحة فقط — يختلف السعر** | `960 ج.م` (مشطوب) ← `940 ج.م 🥇` ← `توفير: 20 ج.م` ← `سعر الشريحة الذهبية` |
| **شريحة فقط — لا يختلف السعر** | `السعر: 960 ج.م` — سعر واحد فقط (لا تكرار) |
| **صفقة اليوم 🔥** | `960 ج.م` (مشطوب) ← `900 ج.م 🔥` ← `توفير: 60 ج.م` ← `صفقة اليوم` |
| **عرض الساعة ⚡** | `960 ج.م` (مشطوب) ← `920 ج.م ⚡` ← `توفير: 40 ج.م` ← `عرض الساعة` |
| **شريحة + عرض معاً** | `960 ج.م` (مشطوب) ← `880 ج.م 🥇🔥` ← `توفير: 80 ج.م` ← `ذهبية + صفقة اليوم` |

#### توضيح مصدر التخفيض

يجب أن يظهر النص التوضيحي أسفل السعر مباشرة:

```
940 ج.م  🥇
سعر الشريحة الذهبية

900 ج.م  🔥
صفقة اليوم

920 ج.م  ⚡
عرض الساعة
```

#### مصفوفة الظهور

| الشاشة | سعر الشريحة 🏷️ | شارة العرض 🔥/⚡ | السعر الأساسي مشطوب | مصدر التخفيض | ملاحظة |
|--------|----------------|-----------------|--------|
| شبكة الشركات | ❌ لا ينطبق | ✅ شارة على الشركة التي عليها عرض | "🔥 إيفا — صفقة اليوم!" |
| منتجات الشركة | ✅ كل منتج بسعر شريحة العميل | ✅ شارة على المنتجات المشمولة | السعر الأصلي مشطوب |
| السلة | ✅ في كل صنف | ✅ في كل صنف مشمول | الخصم في الملخص |
| إنشاء الطلب | ✅ في كل صنف + خصم في الإجمالي | ✅ في كل صنف + خصم في الإجمالي | تفصيل الخصومات |
| تفاصيل الطلب | ✅ سعر الشراء ثابت (مؤرشف) | ✅ شارة فقط (السعر ثابت) | لا تغيير في السعر |

### 7.2 دمج العروض مع الشركات

```
عرض على مستوى النظام:     "صفقة اليوم: خصم 20% على الألبان"
                          ↓
                          كل شركة لديها منتجات ألبان ← تظهر الشارة في منتجاتها

عرض على مستوى شركة:       "عرض شركة إيفا: خصم 15% على كل المنتجات"
                          ↓
                          منتجات إيفا فقط ← تظهر الشارة هناك

عرض الساعة:               "عرض شركة ستارفيل: مكرونة بسعر 15 ج.م"
                          ↓
                          منتج واحد في شركة ستارفيل ← شارة ⚡ على هذا المنتج
```

### 8.1 الاستفادة من groupItems.js الموجود

السلة تستخدم `groupItemsByCompany()` بالفعل — الحل موجود. نحتاج فقط:

1. **تحسين العرض:** كل مجموعة شركة لها Header (اسم + شعار) + مجموع فرعي

```
 │  │  (إذا خارج/داخل السلة)    │               │
 │  │  حالة: ✅ موجود في السلة  │               │
 │  └────────────────────────────┘               │
 │                                               │
 └───────────────────────────────────────────────┘
```

#### 9.3.2 حقل الكمية (Qty Editor)

**المبدأ:** حقل editable بالكامل — ليس مجرد أزرار +/-.

```
┌───────────────────────────┐
│  [-]     [ 17 ]    [+]   │
│          ▲▲▲             │
│    قابل للتحرير المباشر   │
└───────────────────────────┘
```

**السلوك:**

| الإجراء | النتيجة |
|---------|---------|
| ضغط على **+** | الكمية تزيد بمقدار 1 |
| ضغط على **-** | الكمية تنقص بمقدار 1 (لا تقل عن 1) |
| **نقر على الرقم** | يتم تحديد الرقم بالكامل (select all) |
| **كتابة رقم جديد** (مثل 48) | الكمية تصبح 48 فوراً |
| **مسح الرقم وكتابة قيمة** | الجهاز يقبل أي رقم صحيح موجب |
| **التركيز خارج الحقل** (blur) | إذا كانت القيمة فارغة ← تعود إلى 1 |

**لماذا؟** — B2B والجملة تعتمد على كميات كبيرة. الوصول إلى 96 بالضغط على + 95 مرة غير عملي.

#### 9.3.3 زر الإجراء (Action Button)

**المبدأ:** زر واحد يتغير حسب حالة المنتج في السلة.

| حالة المنتج | لون الزر | النص | الإجراء |
|------------|----------|------|---------|
| **غير موجود في السلة** | أخضر / أساسي | **شراء** | يضيف المنتج إلى السلة بالكمية المحددة |
| **موجود في السلة** | أحمر / خطر | **إزالة** | يحذف المنتج من السلة مباشرة (لا تأكيد) |

**بعد الإضافة مباشرة — تأكيد بصري:**

```
┌─────────────────────────────────┐
│  ✅ تمت الإضافة                 │
│  (يظهر لمدة 1.5 ثانية)         │
│  ثم يتحول إلى:                  │
│  [🔴 إزالة]                     │
└─────────────────────────────────┘

مع شارة حالة بجانب الكرت:
✅ موجود في السلة (رقم 3)
```

**لماذا "شراء" بدلاً من "أضف للسلة"؟** — أقصر، أوضح، يناسب طبيعة B2B. المستخدم يشتري، لا يضيف فقط.

#### 9.3.4 قواعد عرض السعر (Pricing Visibility)

**المبدأ:** الشفافية الكاملة — يرى المستخدم مصدر كل رقم.

| الحالة | ما يظهر في الكرت |
|--------|------------------|
| **لا شريحة، لا عرض** | `السعر: 960 ج.م` — سعر واحد فقط |
| **شريحة فقط** (يختلف السعر) | `960 ج.م` (مشطوب) ← `940 ج.م 🥇` ← `توفير: 20 ج.م` |
| **شريحة فقط** (لا يختلف السعر) | `السعر: 960 ج.م` — سعر واحد فقط (لا تكرار) |
| **صفقة اليوم 🔥** | `960 ج.م` (مشطوب) ← `900 ج.م 🔥` ← `توفير: 60 ج.م` ← `صفقة اليوم` |
| **عرض الساعة ⚡** | `960 ج.م` (مشطوب) ← `920 ج.م ⚡` ← `توفير: 40 ج.م` ← `عرض الساعة` |
| **شريحة + عرض معاً** | `960 ج.م` (مشطوب) ← `880 ج.م 🥇🔥` ← `توفير: 80 ج.م` ← `شريحة ذهبية + صفقة اليوم` |

**مثال توضيحي لكل حالة:**

```
┌──────────────────────────┐    ┌──────────────────────────┐
│  بدون خصم               │    │  مع الشريحة فقط          │
│                          │    │                          │
│  السعر: 960 ج.م         │    │  960 ج.م                 │
│                          │    │  940 ج.م  🥇            │
│                          │    │  توفير 20 ج.م           │
│                          │    │  سعر الشريحة الذهبية    │
└──────────────────────────┘    └──────────────────────────┘

┌──────────────────────────┐    ┌──────────────────────────┐
│  مع صفقة اليوم 🔥       │    │  شريحة + عرض معاً       │
│                          │    │                          │
│  960 ج.م                 │    │  960 ج.م                 │
│  900 ج.م  🔥            │    │  880 ج.م  🥇🔥          │
│  توفير 60 ج.م           │    │  توفير 80 ج.م           │
│  صفقة اليوم              │    │  ذهبية + صفقة اليوم     │
└──────────────────────────┘    └──────────────────────────┘
```

#### 9.3.5 قواعد "لا تكرار"

إذا كان `السعر الأساسي === السعر بعد الشريحة`، لا نعرض السعر مرتين:

```
✅ الصحيح:                    ❌ الخطأ:
┌──────────────────┐        ┌──────────────────┐
│  السعر: 960 ج.م │        │  960 ج.م         │
│                  │        │  960 ج.م  🥇    │ ← متطابق!
└──────────────────┘        └──────────────────┘
```

#### 9.4.5 قواعد العرض

| # | القاعدة |
|---|---------|
| 1 | **قيمة الطلب الحالية = أهم عنصر بصري** — أكبر نص في الشريط |
| 2 | **ممنوع عرض عدد الأصناف أو الوحدات** — التركيز على القيمة المالية فقط |
| 3 | **التحديث اللحظي** — كل إضافة/إزالة/تعديل كمية يغير القيمة فوراً |
| 4 | **معلومات الشريحة تظهر فقط عند الحاجة** — لا تظهر إذا لم تختر شريحة |
| 5 | **الضغط على الشريط أو "عرض الطلب"** → ينتقل مباشرة إلى السلة |
| 6 | **إذا السلة فارغة 3 ثوان** → الشريط يختفي تدريجياً |
| 7 | **ارتفاع ثابت ~72px** مع safe-area-bottom |
| 8 | **z-index أعلى من كل العناصر** — يطفو فوق المحتوى |

#### 9.4.6 التفاعل مع Product Card

عند إضافة منتج من خلال Product Card (زر "شراء"):

```
1. المستخدم يضغط [شراء] على المنتج
2. السلة تحدث
3. Floating Bar يتحدث فوراً:
   43,560 ج.م  ←  →  44,500 ج.م  (زيادة بمقدار سعر المنتج)
4. المستخدم يرى التغيير دون مغادرة الشاشة
```

عند حذف منتج (زر "إزالة"):

```
1. المستخدم يضغط [إزالة]
2. Floating Bar يتحدث فوراً:
   44,500 ج.م  ←  →  43,560 ج.م
```

#### 9.4.7 التكامل مع الشرائح

Floating Bar يحتاج أن يعرف:
- `activeTierId` ← الشريحة المحددة حالياً
- `tierMinimum` ← الحد الأدنى للشريحة (مثلاً 50,000 ج.م للذهبية)
- `cartTotal` ← قيمة السلة الحالية
- `tierAchieved` ← boolean: هل `cartTotal >= tierMinimum`

**مصدر البيانات:**
- `activeTierId` من `tiers.js` / session storage
- `tierMinimum` من قائمة الشرائح (API أو مخزنة محلياً)
- `cartTotal` من `cartApi.js` — محسوب لحظياً

#### 9.4.8 الهدف التجاري

```
التاجر/المندوب أثناء التجميع:
"فاتورتي وصلت لكام؟" ← Floating Bar يجيب فوراً
"هل وصلت للشريحة؟"  ← Floating Bar يجيب (إذا كانت الشريحة محددة)
"خلصت؟"              ← [عرض الطلب] ← السلة ← إنشاء طلب
```

## 11. المخاطر والتوصيات

### المخاطر الخاصة بـ Company First

| الخطر | التأثير | الاحتمال | خطة التخفيف |

### التوصيات النهائية

1. **Company First ليس فقط تغيير UI — هو إعادة ترتيب أولويات رحلة المستخدم بالكامل**
2. **الشركات في التبويب الرئيسي (الثاني)** — وليس في قائمة "المزيد"
3. **منتجات الشركة يجب أن تسمح بالبحث الفوري الداخلي** — عبر #search مع فلترة company_id
