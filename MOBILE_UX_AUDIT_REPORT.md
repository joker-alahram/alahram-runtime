## 1. ملخص التنفيذ

### النهج العام

```
Mobile First +  تصميم كروت متجاوبة = إلغاء الجداول التقليدية
```

- **كل `<table>` يتحول إلى `Card List`** مع تخطيط مرن عمودي
- **كل شاشة تُبنى من أعلى إلى أسفل** حسب أولوية المندوب/المشرف: الإجراء → المعلومة
- **الخط الأساسي 16px** مع أحجام لمس لا تقل عن 44px
- **Safe areas** إلزامية لكل الشاشات الثابتة
- **Skeleton loading** لكل شاشة
- **Bottom sheets** هي النمط الوحيد للنوافذ المنبثقة

### القرارات المعمارية (بعد الملاحظات)

| القرار | القيمة |
|--------|--------|
| **المسار الأول (فوري)** | Storefront + Field — Mobile First |
| **المسار الثاني (مؤجل)** | OPS — Desktop First، الجداول باقية |
| **نموذج الأعمال** | **Company First** — الشركات نقطة الدخول، المنتجات داخلها، السلة مجمعة حسبها |
| **محرك البحث** | **#search** الحالي هو المرجع الوحيد + بحث شركات ilike (لا تغيير في RPC أو الفهارس) |
| **الأولوية** | رحلة المستخدم أولاً → الشاشات تبنى حول الرحلة |
| **الشرائح والعروض** | مكونات بيع أساسية — تظهر في كل مكان، خاصة ضمن سياق الشركة |

### مبدأ إعادة التصميم — Company First + Storefront + Field فقط

| من | إلى |
|----|-----|
| **Product First** (منتجات كقائمة مسطحة) | **Company First** (شركات ← منتجات كل شركة ← سلة مجمعة) |
| جدول ب 6 أعمدة (Storefront) | كرت منتج واحد ب 3 أسطر (كود، اسم، سعر/كمية) داخل سياق الشركة |
| صفوف شركة داخل جدول | رأس مجموعة شركة + كروت تحتها |
| 4 طرق بحث مختلفة | محرك #search الموحد + `searchCompanies()` |
| شاشات منفصلة عن الرحلة | شاشات تُبنى حول Commerce Flow (شركة → منتجات → سلة → طلب) |
| الشركات كصفحة ثانوية | الشركات في **التبويب الرئيسي الثاني** في Bottom Nav |
| الفوتر الثابت يختفي مع الكيبورد | الفوتر يتحرك لأعلى تلقائياً |

## 3. P0 — حرج: شاشات الجداول غير القابلة للقراءة

### 3.1 OPS المخزون (inventory/list.js)

**الوضع الحالي:**
- 4 تبويبات (المخزون، المنخفض، الحجوزات، الحركات)
- كل تبويب يستخدم `<table>` مع `overflow-x: auto`
- تبويب المخزون: 7 أعمدة (منتج، وحدة، مخزن، متاح، محجوز، إجمالي، تعديل)
- لا يوجد أي responsive باستثناء الـ scroll

**مشاكل الـ Mobile UX:**
- 7 أعمدة على شاشة 375px = ~53px لكل عمود — غير مقروء
- زر "تعديل" حجمه أصغر من 36px
- لا توجد بطاقات بديلة للموبايل
- عرض الجدول يتطلب تمرير أفقي لكل صف

**التصور الجديد:**

```
┌─────────────────────────┐
│  [بحث وتصفية]           │
│  علامات التبويب (قابلة  │
│  للتمرير)              │
├─────────────────────────┤
│ [تبويب: المخزون]       │
├─────────────────────────┤
│ ┌─── كرت منتج ────────┐│
│ │ كود 1234  مناديل 🧻 ││
│ │ متاح:   500单位      ││
│ │ محجوز:   12          ││
│ │ مخزن القاهرة         ││
│ │ [تعديل]              ││
│ └─────────────────────┘│
│ ┌─── كرت منتج ────────┐│
│ │ كود 5678  صابون 🧼   ││
│ │ متاح:   230单位      ││
│ │ محجوز:    0          ││
│ │ مخزن الإسكندرية      ││
│ │ [تعديل]              ││
│ └─────────────────────┘│
└─────────────────────────┘
```

**التغييرات:**
- إلغاء `<table>` بالكامل
- تحويل كل صف إلى `Card` رأسي: صورة/رمز + كود المنتج + اسم (أعلى)، الكمية المتاحة (وسط، بارز)، المخزن والإجراءات (أسفل)
- شريط فلترة بحث + علامات تبويب أفقية قابلة للتمرير
- زر "تعديل" بحجم لا يقل عن 44px
- Skeleton loading 6 كروت

## 4. P0 — حرج: رحلة المندوب التشغيلية (Workflow)

**ملف:** `domains/field/pages/*` + `domains/storefront/components/activeVisitWorkspace.js`  
**الوضع الحالي:** رحلة المندوب موزعة على Domainين منفصلين (Field + Storefront) مع عدم تناسق في البيانات والإجراءات.

### تدفق العمل الحالي

```
Dashboard → زيارة → إنشاء طلب → متابعة (لا يوجد تسليم)
                                            ╰── المندوب لا يستطيع تأكيد التسليم
```

### المشاكل الحرجة في الـ Mobile UX

| المشكلة | التفاصيل | التأثير |
|---------|----------|---------|
| **لا يوجد تسليم** | المندوب لا يستطيع تأكيد تسليم الطلب. الترانسشن من `dispatched → delivered` يتطلب صلاحيات warehouse/ops | المندوب يفقد متابعة الطلب بعد الإنشاء |
| **كسر سياق الزيارة** | النقر على "إنشاء طلب" ينتقل بالكامل إلى `#checkout` ولا يعود تلقائياً للزيارة | المندوب يضيع في التنقل بين الصفحات |
| **Domain مزدوج** | Field domain و Storefront domain لهما نسختان منفصلتان من الزيارات والطلبات | عدم تناسق في البيانات (localStorage vs DB) |
| **توقيت ثابت (3 خطوات فقط)** | الـ Timeline يظهر فقط (إنشاء، بدء، إنهاء) ولا يظهر الأحداث الفعلية (أمر تم إنشاؤه، تحصيل، ملاحظة) | المندوب لا يرى تاريخ الزيارة الحقيقي |
| **لا يوجد Tracking مستمر** | GPS يتم التقاطه فقط في بداية ونهاية الزيارة. دالة `logLocation` موجودة ولكن غير مستخدمة | عدم وجود مسار حقيقي للزيارة |
| **Confirm() بدائي** | `confirm('إنهاء الزيارة؟')` يستخدم alert المتصفح | تجربة سيئة |
| **Offline غير مدعوم** | Field domain يعتمد كلياً على الشبكة. لا تخزين محلي للزيارات | إذا انقطعت الشبكة، المندوب لا يستطيع العمل |
| **Polling مكثف (30 ثانية)** | الفحص المستمر للزيارة النشطة كل 30 ثانية يستهلك بيانات | استهلاك غير ضروري للبطارية والبيانات |

### تدفق العمل المقترح

```
Dashboard
    │
    ├──> [بدء زيارة] → اختيار عميل → GPS → بدء الزيارة
    │
    ├──> أثناء الزيارة:
    │       ├── [📄 طلب] → Bottom Sheet "إنشاء طلب"
    │       │       ├── اختيار منتجات
    │       │       └── → يعود تلقائياً للزيارة بعد الإنشاء
    │       ├── [💰 تحصيل] → Bottom Sheet "تسجيل تحصيل"
    │       │       ├── المبلغ + طريقة الدفع
    │       │       └── يرتبط بالطلب المحدد
    │       ├── [📝 ملاحظة] → Bottom Sheet "إضافة ملاحظة"
    │       └── [✕ إنهاء] → Bottom Sheet "إنهاء الزيارة"
    │               ├── النتيجة (تمت/ملغية/مؤجلة)
    │               ├── GPS
    │               └── تأكيد
    │
    └──> بعد الزيارة:
            ├── متابعة الطلب: تحديث الحالة في الوقت الحقيقي
            ├── [تأكيد التوصيل] → المندوب يؤكد التسليم مع العميل
            └── [تتبع] → خريطة توضح مسار التوصيل
```

### التغييرات المطلوبة

1. **توحيد الـ Domain**: دمج Field domain و Storefront runtime في مسار واحد (استخدام storefront كنواة + واجهة Field)
2. **Bottom Sheets لكل الإجراءات**: بدلاً من التنقل بين الصفحات، كل إجراء يفتح Bottom Sheet داخل سياق الزيارة
3. **تأكيد التوصيل**: إضافة زر "تأكيد التوصيل" في صفحة تفاصيل الطلب للمندوب — مع التقاط صورة/توقيع العميل
4. **Timeline ديناميكي**: يعرض كل الأحداث (طلب، تحصيل، ملاحظة) وليس 3 خطوات فقط
5. **GPS مستمر**: تفعيل `logLocation` لتسجيل نقاط المسار كل 5 دقائق أثناء الزيارة
6. **Offline First**: تخزين الزيارات محلياً والمزامنة عند توفر الشبكة
7. **إلغاء confirm()**: Bottom Sheet بدلاً من alert
8. **إلغاء polling**: استخدام WebSocket أو Push بدلاً من الاستعلام كل 30 ثانية

**الجهد التقديري:** 5 أيام عمل (أعلى جهد في التقرير)

## 6. P1 — عالي: الشرائح السعرية (Pricing Tiers)

**ملفات:** `domains/storefront/pages/tiers.js`, `domains/storefront/pages/cart.js`, `domains/ops/pages/pricing.js`, `services/storefront/cartApi.js`, `services/storefront/productsApi.js`  
**الوضع الحالي:** نظام الشرائح موجود في قاعدة البيانات (pricing_tiers, product_prices, customer_tier_assignments) مع عرض محدود في الواجهة.

### المشاكل الحرجة

| المشكلة | التفاصيل | التأثير |
|---------|----------|---------|
| **لا يوجد منتقي شرائح** | المستخدم لا يستطيع رؤية شريحته الحالية أو تغييرها | تجربة عمياء |
| **مندوب → عميل → يفقد الشريحة** | عندما يختار المندوب عميلاً، `_fetchCustomerTierCode()` لا يجلب شريحة العميل لأن `actor.type !== 'customer'` | العميل لا يحصل على سعر شريحته |
| **لا يظهر في الشيك أوت** | صفحة إنشاء الطلب لا تظهر وسام الشريحة لكل صنف | المستخدم لا يعرف السعر المستخدم |
| **لا يظهر في PDF** | `pdfService.js` لا يتضمن معلومات الشريحة | الفاتورة بدون إشارة للشريحة |
| **TIER_CHANGED غير مستخدم** | الحدث `TIER_CHANGED` معرف في Event Bus ولكن لا يصدره أي كود | stale detection غير موثوق |
| **لون الشريحة غير مستخدم** | `tier_color` موجود في DB ولكن غير مستخدم في الـ UI | إمكانية بصرية مهدرة |
| **Color code** | Not used in UI | Missed visual opportunity |

### 11.1 Storefront العملاء (customers.js)

**التحسينات:** كروت أكبر مع تأثيرات انتقالية أفضل، تحسين النصوص القصيرة.

### 11.3 Portal الطلبات/الزيارات

**التحسينات:** تحويل أي جدول صغير إلى كروت.

### 11.4 جميع Modals

- **showModal()**: يجب أن تكون `Bottom Sheet` دائماً — وليس `overlay` عادي مع form
- **confirmDelete()**: يجب أن يكون Bottom Sheet مع زر أحمر واضح
- **Native alert/confirm**: يجب استبدالها بـ Bottom Sheet أو Toast

## 13. ملاحظات إضافية — القرارات المعمارية

### الوحدات الجديدة التي تمت إضافتها

بناءً على طلبك، تم إضافة 5 وحدات تشغيلية وتجارية إلى نطاق المراجعة:

| الوحدة | القسم | الأولوية | الجهد | الملخص |
|--------|-------|----------|-------|--------|
| **رحلة المندوب التشغيلية** | Field + Storefront | **P0** | 5 أيام | توحيد الـ Domain، Bottom Sheets لكل إجراء، إضافة تسليم، GPS مستمر، Offline |
| **الشرائح السعرية (Tiers)** | Storefront + OPS | **P1** | 4 أيام | منتقي شرائح Bottom Sheet، إظهار في كل الشاشات، إصلاح شريحة العميل عبر المندوب |
| **صفقة اليوم (Daily Deal)** | Storefront + OPS | **P1** | 3 أيام | هوية بصرية مستقلة 🔥، Countdown حقيقي، تكامل مع السلة |
| **عرض الساعة (Flash Offer)** | Storefront + OPS | **P1** | 3 أيام | عداد حقيقي، حالة بألوان، تكامل مع السلة + الفاتورة |
| **تصفح المنتجات** | Storefront | **P1** | 4 أيام | بحث فوري، بطاقات محسّنة، عروض بارزة، تحميل المزيد |

### القرارات المعمارية المعتمدة

1. **مبدأ المسارين:** Storefront + Field = Mobile First | OPS = Desktop First مؤقتاً
2. **Company First Commerce:** الشركات هي نقطة الدخول الأساسية — المنتجات داخلها — السلة مجمعة حسبها — البحث يبدأ بالشركات
3. **محرك البحث الموحد:** #search الحالي + `searchCompanies()` (ilike على companies) — لا تغيير في RPC أو الفهارس
4. **Commerce Flow أولاً:** الشاشات تُبنى حول رحلة العميل (شركة → منتجات → سلة → طلب) والمندوب (زيارة → شركة → منتجات → طلب)
5. **الشرائح والعروض:** مكونات بيع أساسية — تظهر ضمن سياق الشركة (شركة عليها عرض) والمنتج (سعر حسب الشريحة + شارة العرض)
6. **لا نكسر الموجود:** نغلف ونعيد استخدام، لا نستبدل

### الخطوات التالية الموصى بها:

1. **اعتماد هذا التقرير + العمارة الجديدة** (في `MOBILE_STOREFRONT_UX_ARCHITECTURE.md`)
2. **Phase 0: التصميم** — Wireframes لرحلة العميل + رحلة المندوب قبل أي كود
3. **بالتوازي: إنشاء Core CSS** `mobile-2026.css` + مكون البحث الموحد
4. **Phase 1: Core Storefront** — الرئيسية + المنتجات + السلة + إنشاء الطلب
5. **اختبار على أجهزة حقيقية** بعد كل شاشة

### المبادئ التي يجب تطبيقها في الكود الجديد:

